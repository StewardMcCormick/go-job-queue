# Инфраструктурный Job Queue/Task Scheduler сервис + SDK к нему

## Термины:
- Worker - внешний сервис-обработчик задач
- Scheduler - сущность, отвечающая за планирование исполнения задач - дедлайны, очереди, графы зависимостей
- Dispatcher - сущность, отвечающая за исполнение задач - передача их воркерам, следит за воркерами, которые долго 
  выполняют свои задачи (отменять по timeout?)

## Стэк:
- Сеть - gRPC (в итоге скроется за SDK), gRPC server-streaming RPC/server-client?, grcp-gateway:
  - Воркер открывает stream соединение, dispatcher отсылает ему задачи (нужно задавать длину стрима, чтобы его не 
    перегружать? - воркер сообщает, сколько задач можно поставить ему в очередь, если она переполняется - ждать), 
    когда воркер завершает работу - отменяет контекст стрима => стрим закрывается на стороне сервера (что будет, если 
    воркер упадет? добавить timeout на задачу - настроенную или дефолтную)
- Хранение данных: Kafka (тунель между планировщиком и диспетчером), Postgres (инфа о выполненных задачах) + goose 
  (миграции), Redis (информация о текущих задачах на всем цикле их жизни, чтобы быстрее получать статистику) - работает 
  в режиме Append Only File
- Логи - uber-zap
- Много конкурентности

## Фичи:
- Возможность установить дедлайн задач - сервис гарантировано передаст на выполнение воркеру в назначенное время
  - В случае, если зависящая задача имеет дедлайн, он устанавливается для всей группы, иначе максимальный из дедлайнов 
    задач-зависимостей
  - Возможность настроить deadline policy:
    - Fail - провалена вся группа, задачи завершаются без retry
    - Retry - задачи ставятся на исполнение повторно. Количество попыток можно настроить
- Любой формат payload - ответственность за валидацию, корректность данных лежит на пользователе
- Возможность задавать задачам зависимость от других задач (по id) - в формате списка задач-зависимостей
  - При таком запросе во всех зависимых задачах устанавливаются задачи, которые от них зависят - нужно для deadline 
    policy
- SDK
- Воркер взаимодействует с диспетчером по server-stream RPC
  - Все ручки дублируются через grpc-gateway в HTTP ручки
- Получение статистики:
  - Длина очереди по типам задач
  - Количество активных воркеров
  - Процент успешных/упавших/истекших/отмененных задач
  - Среднее время выполнения задач

## Цикл жизни задачи:
- Created - задача создана.
- Pending - задача готова к исполнению - ждет очереди/дедлайна исполнения
  - Retry - задача повторно поставлена в очередь на исполнение после ошибки
  - Failed - задача провалена после того, как кончились попытки retry
  - Cancelled - задача отменена пользователем (через отдельный запрос?) - если отменена задача, входящая в группу, 
    отменяется вся группа => нужно в каждой задаче хранить, какие задачи от нее зависят. Отмена Running задач не 
    поддерживается
- Running - запущена в обработку
- Completed - задача завершена

- Дополнительный статус -  MissedDeadline - пользователь сам настраивает, будет ли после этого исполняться задача 
  или dispatcher должен ее пропустить

- При создании задачи она сохраняется в redis, обновляется по мере прохождения по циклу жизни. вся информация о 
  незавершенных задачах достается из Redis, после завершения - кладется в postgres, из redis удаляется (можно периодически выгружать данные из redis в postgres для консистентности и гарантии сохранности данных - делать конкурентно?)

##  Типы приоритетов:
- Background
- Normal
- High
- Immediate

- Задачи, дедлайн который наступил помечаются как ***Immediate***.
- Если поступила задача, имеющая зависимости и дедлайн - они все помечаются как ***Immediate***
- Все задачи по типам распределяются планировщиком по соответствующим топикам Kafka
- Dispatcher обходит топики в следующем порядке:
```
Immediate ──▶ High ──▶ Normal ──▶ Background
```
- Как обезопаситься от "зависания" менее приоритетных задач и нужно ли это? round-robin - определить порядок и 
  проценты выполнения задач

## Жизненный цикл воркера
- Registered - воркер зарегистрирован в dispatcher
- Working - исполняет задачи
- Dead - воркер умер (не отвечает на запросы/heartbeat/не смог исполнить задачу после нескольких ретраев?)
- Unregistered - воркер снялся с исполнения по запросу

## Domain Layer:
```protobuf
syntax = "proto3";

import "google/protobuf/timestamp.proto";

service JobQueueService {
  rpc Health(HealthRequest) returns (HealthResponse);

  rpc CreateTask(CreateTaskRequest) returns (CreateTaskResponse);
}

// ================Request Models================

message HealthRequest {
  int32 num = 1;
}

message CreateTaskRequest {
  string priority = 3;
  string type = 4;
  map<string, bytes> payload = 5;
  uint32 retry_number = 6;
  google.protobuf.Timestamp deadline = 7;
  repeated string depends_on = 8;
}

// ================Domain Models================

message Task {
  string id = 1;
  TaskStatus status = 2;
  TaskPriority priority = 3;
  string type = 4;
  map<string, bytes> payload = 5;
  uint32 should_retry_number = 6;
  uint32 retries = 7;
  google.protobuf.Timestamp deadline = 8;
  repeated string depends_on = 9;
  repeated string dependency_for = 10;
  google.protobuf.Timestamp created_at = 11;
  google.protobuf.Timestamp updated_at = 12;
  google.protobuf.Timestamp started_at = 13;
  google.protobuf.Timestamp completed_at = 14;
}

message Worker {
  string id = 1;
  string addr = 2;
  string task_type = 3;
  uint32 concurrency = 4;
  WorkerStatus status = 5;
}

enum TaskStatus {
  TASK_STATUS_UNSPECIFIED = 0;
  TASK_STATUS_CREATED = 1;
  TASK_STATUS_PENDING = 2;
  TASK_STATUS_RETRY = 3;
  TASK_STATUS_FAILED = 4;
  TASK_STATUS_CANCELLED = 5;
  TASK_STATUS_RUNNING = 6;
  TASK_STATUS_COMPLETED = 7;
}

enum TaskPriority {
  TASK_PRIORITY_UNSPECIFIED = 0;
  TASK_PRIORITY_BACKGROUND = 1;
  TASK_PRIORITY_NORMAL = 2;
  TASK_PRIORITY_HIGH = 3;
  TASK_PRIORITY_IMMEDIATE = 4;
}

enum WorkerStatus {
  WORKER_STATUS_UNSPECIFIED = 0;
  WORKER_STATUS_REGISTERED = 1;
  WORKER_STATUS_WORKING = 2;
  WORKER_STATUS_DEAD = 3;
  WORKER_STATUS_UNREGISTERED = 4;
}

// ================Response Models================

message HealthResponse {
  int32 repeated_num = 1;
}

message CreateTaskResponse {
  Task task = 1;
}
```
