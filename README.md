# Инфраструктурный Job Queue/Task Scheduler сервис + SDK к нему

## Термины:
- Worker - внешний сервис-обработчик задач
- Scheduler - сущность, отвечающая за планирование исполнения задач - дедлайны, очереди, графы зависимостей
- Dispatcher - сущность, отвечающая за исполнение задач - передача их воркерам, следит за воркерами, которые долго выполняют свои задачи (отменять по timeout?)

## Стэк:
- Сеть - gRPC (в итоге скроется за SDK), gRPC server-streaming RPC/server-client?, grcp-gateway:
  - Воркер открывает stream соединение, dispatcher отсылает ему задачи (нужно задавать длину стрима, чтобы его не перегружать? - воркер сообщает, сколько задач можно поставить ему в очередь, если она переполняется - ждать), когда воркер завершает работу - отменяет контекст стрима => стрим закрывается на стороне сервера (что будет, если воркер упадет? добавить timeout на задачу - настроенную или дефолтную)
- Хранение данных: Kafka (тунель между планировщиком и диспетчером), Postgres (инфа о выполненных задачах) + goose (миграции), Redis (информация о текущих задачах на всем цикле их жизни, чтобы быстрее получать статистику) - работает в режиме Append Only File
- Логи - uber-zap
- Много конкурентности

## Фичи:
- Возможность установить дедлайн задач - сервис гарантировано передаст на выполнение воркеру в назначенное время
  - В случае, если зависящая задача имеет дедлайн, он устанавливается для всей группы, иначе максимальный из дедлайнов задач-зависимостей
  - Возможность настроить deadline policy:
    - Fail - провалена вся группа, задачи завершаются без retry
    - Retry - задачи ставятся на исполнение повторно. Количество попыток можно настроить
- Любой формат payload - ответственность за валидацию, корректность данных лежит на пользователе
- Возможность задавать задачам зависимость от других задач (по id) - в формате списка задач-зависимостей
  - При таком запросе во всех зависимых задачах устанавливаются задачи, которые от них зависят - нужно для deadline policy
- SDK
- Воркер взаимодействует с диспетчером по server-stream RPC
  - Все ручки дублируются через grpc-gateway в HTTP ручки
- Получение статистики:
  - Длина очереди по типам задач
  - Количество активных воркеров
  - Процент успешных/упавших/истекших/отмененных задач
  - Среднее время выполнения задач

## Цикл жизни задачи:
- Created - задача создана.
- Pending - задача готова к исполнению - ждет очереди/дедлайна исполнения
  - Retry - задача повторно поставлена в очередь на исполнение после ошибки
  - Failed - задача провалена после того, как кончились попытки retry
  - Cancelled - задача отменена пользователем (через отдельный запрос?) - если отменена задача, входящая в группу, отменяется вся группа => нужно в каждой задаче хранить, какие задачи от нее зависят. Отмена Running задач не поддерживается
- Running - запущена в обработку
- Completed - задача завершена

- Дополнительный статус -  MissedDeadline - пользователь сам настраивает, будет ли после этого исполняться задача или dispatcher должен ее пропустить

- При создании задачи она сохраняется в redis, обновляется по мере прохождения по циклу жизни. вся информация о незавершенных задачах достается из Redis, после завершения - кладется в postgres, из redis удаляется (можно периодически выгружать данные из redis в postgres для консистентности и гарантии сохранности данных - делать конкурентно?)

##  Типы приоритетов:
- Background
- Normal
- High
- Immediate

- Задачи, дедлайн который наступил помечаются как ***Immediate***.
- Если поступила задача, имеющая зависимости и дедлайн - они все помечаются как ***Immediate***
- Все задачи по типам распределяются планировщиком по соответствующим топикам Kafka
- Dispatcher обходит топики в следующем порядке:
```
Immediate ──▶ High ──▶ Normal ──▶ Background
```
- Как обезопаситься от "зависания" менее приоритетных задач и нужно ли это? round-robin - определить порядок и проценты выполнения задач

## Жизненный цикл воркера
- Registered - воркер зарегистрирован в dispatcher
- Working - исполняет задачи
- Dead - воркер умер (не отвечает на запросы/heartbeat/не смог исполнить задачу после нескольких ретраев?)
- Unregistered - воркер снялся с исполнения по запросу

## Domain Layer:
```Go
type TaskType string

type Priority string

const (
  PriorityImmediate  Priority = "immediate"
  PriorityHigh       Priority = "high"
  PriorityNormal     Priority = "normal"
  PriorityBackground Priority = "background"
)

type TaskPayload map[string]interface{}

type TaskStatus string

const (
  TaskStatusCreated        TaskStatus = "created"
  TaskStatusPending        TaskStatus = "pending"
  TaskStatusRetry          TaskStatus = "retry"
  TaskStatusFailed         TaskStatus = "failed"
  TaskStatusCancelled      TaskStatus = "cancelled"
  TaskStatusRunning        TaskStatus = "running"
  TaskStatusCompleted      TaskStatus = "completed"
  TaskStatusMiddesDeadline TaskStatus = "missed_deadline"
)

type WorkerStatus string

const (
  WorkerStatusRegistered   WorkerStatus = "registered"
  WorkerStatusWorking      WorkerStatus = "working"
  WorkerStatusDead         WorkerStatus = "dead"
  WorkerStatusUnregistered WorkerStatus = "unregistered"
)

type Task struct {
  Id                 string
  Status             TaskStatus
  Payload            TaskPayload
  Type               TaskType
  Priority           Priority
  RetryNumber        uint8
  Deadline           *time.Time
  RetryAfterDeadline bool
  Timeout            time.Duration
  DependsOn          []string
  DependencyFor      []string
  CreatedAt          time.Time
  UpdatedAt          time.Time
  StartedAt          time.Time
  CompletedAt        time.Time
}

type Worker struct {
  Id          string
  Addr        string
  ForTaskType TaskType
  Concurrency uint32
  Stream      grpc.ServerStream
  Status      WorkerStatus
}
```
