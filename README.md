# Инфраструктурный Job Queue/Task Scheduler сервис

## Термины:
- Worker - внешний сервис-обработчик задач
- Scheduler - сущность, отвечающая за планирование исполнения задач - дедлайны, очереди, графы зависимостей
- Dispatcher - сущность, отвечающая за исполнение задач - передача их воркерам, следит за воркерами, которые долго выполняют свои задачи (отменять по timeout?)

## Стэк:
- Сеть - gRPC (в итоге скроется за SDK), gRPC long-polling, grcp-gateway
- Хранение данных: Kafka (тунель между планировщиком и диспетчером), Postgres (инфа о выполненных задачах) + goose (миграции), Redis (информация о текущих задачах на всем цикле их жизни, чтобы быстрее получать статистику) - работает в режиме Append Only File
- Логи - uber-zap
- Много конкурентности

## Фичи:
- Возможность установить дедлайн задач - сервис гарантировано передаст на выполнение воркеру в назначенное время
    - В случае, если зависящая задача имеет дедлайн, он устанавливается для всей группы, иначе максимальный из дедлайнов задач-зависимостей
    - Возможность настроить deadline policy:
        - Fail - провалена вся группа, задачи завершаются без retry
        - Retry - задачи ставятся на исполнение повторно. Количество попыток можно настроить
- Любой формат payload - ответственность за валидацию, корректность данных лежит на пользователе
- Возможность задавать задачам зависимость от других задач (по id) - в формате списка задач-зависимостей
    - При таком запросе во всех зависимых задачах устанавливаются задачи, которые от них зависят - нужно для deadline policy
- SDK
- Воркер взаимодействует с диспетчером по pull модели - периодически опрашивает его по gRPC long-polling (30-60 секунд):
    - Все ручки дублируются через grpc-gateway в HTTP ручки
- Получение статистики:
    - Длина очереди по типам задач
    - Количество активных воркеров
    - Процент успешных/упавших/истекших/отмененных задач
    - Среднее время выполнения задач

## Цикл жизни задачи:
- Created - задача создана.
- Pending - задача готова к исполнению - ждет очереди/дедлайна исполнения
    - Retry - задача повторно поставлена в очередь на исполнение после ошибки
    - Failed - задача провалена после того, как кончились попытки retry
    - Cancelled - задача отменена пользователем (через отдельный запрос?) - если отменена задача, входящая в группу, отменяется вся группа => нужно в каждой задаче хранить, какие задачи от нее зависят. Отмена Running задач не поддерживается
- Running - запущена в обработку
- Completed - задача завершена

- Дополнительный статус -  MissedDeadline - пользователь сам настраивает, будет ли после этого исполняться задача или dispatcher должен ее пропустить

- При создании задачи она сохраняется в redis, обновляется по мере прохождения по циклу жизни. вся информация о незавершенных задачах достается из Redis, после завершения - кладется в postgres, из redis удаляется (можно периодически выгружать данные из redis в postgres для консистентности и гарантии сохранности данных - делать конкурентно?)

## Примерный принцип работы

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  Сервис А       │     │   Job Queue     │     │  Сервис Б       │
│  (клиент)       │────▶│    (ядро)       │◀────│  (воркер)       │
│                 │     │                 │     │                 │
│ "выполни задачу │     │ сохраняет,      │     │ забирает задачу,│
│  типа X"        │     │ планирует,      │     │ делает работу,  │
│                 │     │ ретраит         │     │ сообщает статус │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```
